Anatomy of a tuple — display on projector
Tuples look like lists but use ( ) and can never be changed after creation
point = (10, 25) colour = (255, 128, 0) print(point[0]) # 10 point[0] = 99 # TypeError!
( ) — parentheses. Optional but strongly recommended for clarity. The comma is what actually makes a tuple.
Indexing works exactly like lists — point[0], colour[-1], slices — all supported.
Immutable — once created, no item can be added, removed, or changed. Any attempt raises a TypeError.
Ordered — maintains insertion order, supports indexing and slicing like a list.
Hashable — can be used as a dictionary key, unlike lists.

Example 1 — Creating tuples, indexing, the comma rule
main.py
# Creating tuples
point   = (10, 25)
colour  = (255, 128, 0)
person  = ("Aisha", 15, "A")
empty   = ()

# Indexing — same as lists
print(point[0])        # 10
print(colour[-1])      # 0
print(person[1:3])     # (15, 'A')

# The comma makes the tuple — not the parentheses!
not_tuple = (42)       # int — just brackets for grouping
is_tuple  = (42,)      # tuple — the trailing comma matters
print(type(not_tuple)) # <class 'int'>
print(type(is_tuple))  # <class 'tuple'>

# Parentheses are optional
implicit = 1, 2, 3
print(implicit)          # (1, 2, 3) — Python adds parens
print(type(implicit))    # <class 'tuple'>
Output

0
(15, 'A')
<class 'int'>
<class 'tuple'>
(1, 2, 3)
<class 'tuple'>
🚨The single-element tuple trap: (42) is just the integer 42 in brackets — not a tuple. You must write (42,) with a trailing comma. Run both live and show type() — this surprises every beginner without exception.
Example 2 — Immutability: what breaks, what still works
main.py
t = (1, 2, 3)

# ✓ These all work — reading is fine
print(t[0])           # 1
print(len(t))          # 3
print(2 in t)          # True
print(t[1:])           # (2, 3)
print(t + (4, 5))      # (1, 2, 3, 4, 5) — new tuple
print(t * 2)           # (1, 2, 3, 1, 2, 3) — new tuple

# ✗ These raise errors — no modification allowed
t[0] = 99             # TypeError!

Example 3 — Tuple unpacking
main.py
# Basic unpacking — assign multiple variables at once
point        = (10, 25)
x, y         = point
print(f"x={x}, y={y}")     # x=10, y=25

# Unpack a person tuple
person       = ("Aisha", 15, "A")
name, age, grade = person
print(f"{name} is {age} with grade {grade}")

# The variable-swap trick — no temp variable needed!
a, b = 10, 20
print(f"Before: a={a}, b={b}")
a, b = b, a
print(f"After:  a={a}, b={b}")

# Unpack with _ to discard unwanted values
name, _, grade = ("Omar", 16, "B")
print(f"{name}: {grade}")  # age discarded with _
Output
x=10, y=25
Aisha is 15 with grade A
Before: a=10, b=20
After: a=20, b=10
Omar: B
💡The swap trick always gets a reaction: Before running a, b = b, a ask students to predict what happens. Most expect it to not work (they're used to needing a temp variable). Show them it does — then explain: Python evaluates the right side first as a tuple (b, a) = (20, 10), then unpacks it into a and b.
Example 4 — Three use cases where tuples are the right choice
main.py
# Use case 1: Function returning multiple values
def min_max(numbers):
    """Return (min, max) of a list."""
    return min(numbers), max(numbers)   # implicit tuple

lo, hi = min_max([5, 2, 8, 1, 9])
print(f"Min={lo}, Max={hi}")   # Min=1, Max=9

# Use case 2: Fixed data — RGB colour, GPS coordinate
RED   = (255, 0, 0)
GREEN = (0, 255, 0)
home  = (33.7, 73.1)    # latitude, longitude

# Use case 3: Tuple as dictionary key (lists can't do this!)
grid = {}
grid[(0, 0)] = "origin"
grid[(1, 2)] = "point A"
grid[(3, 4)] = "point B"
print(grid[(1, 2)])      # "point A"

# ✗ List as dict key — TypeError!
grid[[0, 0]] = "fail"   # TypeError: unhashable type: 'list'
Output
Min=1, Max=9
point A
TypeError: unhashable type: 'list'
🔑Why tuples can be dict keys but lists can't: Dict keys must be hashable — they need a fixed identity. Tuples are immutable so their hash never changes. Lists can change, so their hash would change, which would break the dict. This is why (0,0) works as a key but [0,0] doesn't.

Starter code
main.py — starter
def place(grid, x, y, label):
    """Add a labelled point to the grid dict."""
    grid[(x, y)] = label

def show_grid(grid):
    """Print all points with their labels."""
    print("Grid contents:")
    for (x, y), label in grid.items():
        print(f"  ({x}, {y}) → {label}")

def find_point(grid, x, y):
    """Return label at (x,y) or 'empty' if not found."""
    return grid.get((x, y), "empty")

def main():
    grid = {}
    place(grid, 0, 0, "origin")
    place(grid, 1, 2, "school")
    place(grid, 3, 4, "park")
    show_grid(grid)
    print(find_point(grid, 1, 2))    # "school"
    print(find_point(grid, 9, 9))    # "empty"

💡How it works behind the scenes: Python evaluates the right side first, creating an implicit tuple (b, a) = (20, 10). It then unpacks this tuple into the left side variables: a = 20, b = 10. The whole thing happens atomically — no temporary variable is ever needed.
Preview — named tuples (collections module)

Regular tuples use position — point[0] could mean x or y. Python's namedtuple gives each position a name, making tuples as readable as dicts while staying immutable. You'll encounter this in real Python code — here's a preview.

Regular tuple vs named tuple — same immutability, much more readable
⚠ Regular tuple
p = (10, 25)
print(p[0])  # x? y?
print(p[1])  # must remember!
Position-based — easy to mix up x and y.
✓ Named tuple
from collections import namedtuple
Point = namedtuple("Point", ["x", "y"])
p = Point(10, 25)
print(p.x)   # 10 — obvious!
print(p[0])  # also works
Still immutable. Access by name or index.
💡Don't teach namedtuple today — this is a preview only. Mention it exists, show the syntax briefly, and move on. Students who want to explore it have a clear hook for self-study. The key lesson remains: plain tuples for simple grouped immutable data.

Key vocabulary for this lesson
TupleAn immutable, ordered sequence created with (). Elements cannot be added, removed, or changed after creation.
ImmutableCannot be changed after creation. Tuples are immutable; lists and dicts are mutable.
Tuple unpackingAssigning multiple variables from a tuple in one line: x, y = (10, 25). The number of variables must match the tuple length.
HashableA type whose value never changes, allowing it to be used as a dict key. Tuples are hashable; lists are not.
Single-element tuple(42,) — the trailing comma is essential. Without it, (42) is just the integer 42 in brackets.
Implicit tuplePython creates a tuple automatically when you write comma-separated values without brackets: a = 1, 2, 3 creates a tuple.

Teacher notes
🚨The single-element tuple trap — demonstrate it every time: (42) is an int; (42,) is a tuple. Run print(type((42))) and print(type((42,))) side by side. The comma is what makes a tuple. Students who skip the comma spend 20 minutes debugging mysterious type errors.
⚠️Common confusion — when to use tuple vs list: The decision rule is: if the data has a fixed structure with a specific meaning for each position (x and y, name and age), use a tuple. If the data is a growing collection of same-type items, use a list. This is worth repeating three times across the lesson.
⚠️Unpacking count must match: a, b = (1, 2, 3) raises a ValueError — too many values to unpack. Students who try this will see a confusing error. Show it live and explain: the number of variables on the left must exactly match the tuple length.
💡Connect to what students already know: for k, v in d.items() is tuple unpacking. a, b = get_numbers() from Lesson 19 is tuple unpacking. The swap trick a, b = b, a is tuple packing + unpacking. Students have been using tuples all course — this lesson gives the name to what they already do.
💡Activity 2 timing: The data structure chooser is the lesson's most conceptually demanding activity. If the class is struggling, do the first 4 scenarios together as a group, then let students try the last 4 independently. The justification is as important as the answer — require one sentence of reasoning for each choice.
