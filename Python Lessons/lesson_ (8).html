Indentation level visualiser — show on projector
Each colour stripe = one nesting level. Code at that indent level belongs to that block.
💡Draw this on the board as nested boxes: Level 0 is the outer box. Level 1 is a box drawn inside it. Level 2 is a box drawn inside that. The indentation in code maps exactly to which box you're inside. Students who struggle with indentation visualise it as boxes, not spaces.
Nested if vs compound and — choose the cleaner one
✗ Unnecessarily nested — use and instead
if age >= 18:
    if has_id == "yes":
        print("Entry allowed")
    else:
        print("Need ID")
else:
    print("Need ID")
Two identical else branches — repeated code. The outer else and inner else say the same thing.
✓ Better — compound and is cleaner
if age >= 18 and has_id == "yes":
    print("Entry allowed")
else:
    print("Need ID")
One condition, one else. Less code, same logic, far more readable.
✗ Flat and — can't give separate feedback per field
if user == "admin" and pw == "1234":
    print("Welcome")
else:
    print("Login failed")
# Can't tell user WHICH field was wrong
The else can't distinguish "wrong username" from "wrong password" — poor UX.
✓ Better — nested gives specific error messages
if user == "admin":
    if pw == "1234":
        print("Welcome")
    else:
        print("Wrong password")
else:
    print("Unknown username")
Specific messages: "wrong password" vs "unknown username". This is when nesting adds value.
🎯The decision rule: Use and when both conditions lead to the same else. Use nested if when each condition needs its own separate response.
Live demo — code examples
Example 1 — Two-level nesting: ticket and bag check
main.py
has_ticket = input("Do you have a ticket? (yes/no): ")

if has_ticket == "yes":                   # Level 1
    bag_kg = float(input("Bag weight in kg: "))
    if bag_kg <= 23:                       # Level 2
        print("✓ Bag accepted. Proceed to gate.")
    else:                                  # Level 2 else
        print(f"✗ Bag too heavy ({bag_kg}kg). Max is 23kg.")
else:                                      # Level 1 else
    print("✗ No ticket. Please buy one first.")
Three possible paths
ticket=yes, bag=18 → ✓ Bag accepted. Proceed to gate.
ticket=yes, bag=27 → ✗ Bag too heavy (27.0kg). Max is 23kg.
ticket=no → ✗ No ticket. Please buy one first.
Example 2 — Three-level nesting: ATM simulation
main.py
pin     = input("Enter PIN: ")
balance = 5000

if pin == "1234":                              # Level 1: PIN check
    print(f"Welcome! Balance: Rs {balance}")
    amount = float(input("Withdrawal amount: Rs "))

    if amount > 0:                              # Level 2: positive amount
        if amount <= balance:                   # Level 3: sufficient funds
            balance -= amount
            print(f"✓ Rs {amount:.0f} dispensed. New balance: Rs {balance}")
        else:
            print("✗ Insufficient funds.")
    else:
        print("✗ Amount must be greater than zero.")
else:
    print("✗ Incorrect PIN. Card retained.")
Four possible paths
pin=1234, amount=2000 → ✓ Rs 2000 dispensed. New balance: Rs 3000
pin=1234, amount=9000 → ✗ Insufficient funds.
pin=1234, amount=-50 → ✗ Amount must be greater than zero.
pin=9999 → ✗ Incorrect PIN. Card retained.
💡Three levels is typically the limit. Beyond three levels of nesting, code becomes very hard to read and debug. If students find themselves at level 4+, it's a sign the logic should be refactored into functions (Week 4 topic — plant the seed now).
Example 3 — Input validation: crash vs safe
main.py — broken: no validation
score = int(input("Enter score: "))   # crashes if user types "abc"
if score >= 50:
    print("Pass")
else:
    print("Fail")
main.py — safe: validated with .isdigit()
raw = input("Enter score (0–100): ")

if raw.isdigit():                     # check: is it a valid whole number?
    score = int(raw)                   # safe to convert now
    if 0 <= score <= 100:             # check: is it in range?
        if score >= 50:
            print("Pass")
        else:
            print("Fail")
    else:
        print("Score must be between 0 and 100.")
else:
    print("Please enter a number, not letters.")
All four cases handled cleanly
Input: 75 → Pass
Input: 30 → Fail
Input: 150 → Score must be between 0 and 100.
Input: abc → Please enter a number, not letters.
⚠️Limitation of .isdigit(): It returns False for negative numbers ("-5") and decimals ("3.14") because - and . are not digits. For now this is fine — we'll handle decimals and negatives properly with try/except in Week 6.

Activity 1 — Login simulator
Activity 1 · 15 minutes · Individual
Build a username + password checker with nested conditions and specific error messages
Students build a complete two-field login checker. The outer condition checks the username; only if correct does it check the password. Each failure gives a specific, helpful message. Validation ensures neither field is left blank.
  1. Ask for username and password using input()
  2. First validate: if either field is empty (== ""), print "Fields cannot be blank" and stop
  3. If both fields have content, check username with outer if — print "Unknown user" if wrong
  4. If username is correct, check password with inner if — print "Wrong password" if wrong
  5. If both correct, print a personalised welcome with the username in an f-string
  6. Extension: add an account_locked variable — if True, reject login before checking anything else
Starter code
main.py — starter
# Login Simulator
CORRECT_USER = "admin"      # store expected credentials
CORRECT_PASS = "python123"

print("=== Login ===")
username = input("Username: ")
password = input("Password: ")

# Step 1: validate — neither field blank
if username == "" or password == "":
    print("✗ Fields cannot be blank.")
else:
    # Step 2: check username
    if username == CORRECT_USER:
        # Step 3: check password (only reached if username correct)
        if password == ???:
            print(f"✓ Welcome, {???}!")
        else:
            print("✗ {???}")
    else:
        print("✗ {???}")
All five cases to test
user="", pass="" → ✗ Fields cannot be blank.
user="admin", pass="" → ✗ Fields cannot be blank.
user="hacker", pass=any → ✗ Unknown username.
user="admin", pass="xyz" → ✗ Wrong password.
user="admin", pass="python123" → ✓ Welcome, admin!
Activity 2 — Safe divider
Activity 2 · 8 minutes · Individual
Validate two numbers before dividing — handle letters and zero denominator
Students build the safest possible version of the division calculator from Lesson 3. Uses .isdigit() to validate before converting, then checks the denominator is not zero before dividing. Connects Lessons 3, 4, and 8.
  1. Read both inputs as raw strings (no int() yet)
  2. Use .isdigit() on both — if either fails, print "Please enter whole numbers only"
  3. If both are digits, convert to int and check if denominator is zero
  4. If denominator is zero, print "Cannot divide by zero"
  5. Otherwise, compute and print the result using :.4f formatting
Starter code
main.py — starter
raw_a = input("Enter numerator: ")
raw_b = input("Enter denominator: ")

if raw_a.isdigit() and raw_b.isdigit():
    a = int(raw_a)
    b = int(raw_b)
    if b == ???:
        print("✗ Cannot divide by zero.")
    else:
        print(f"{a} ÷ {b} = {???:.4f}")
else:
    print("✗ Please enter whole numbers only.")
Test cases
a=10, b=3 → 10 ÷ 3 = 3.3333
a=10, b=0 → ✗ Cannot divide by zero.
a=abc, b=3 → ✗ Please enter whole numbers only.
a=10, b=xyz → ✗ Please enter whole numbers only.
Check for understanding — exit ticket
In the login simulator, why is the password check placed inside the username if block rather than as a separate outer if?
ABecause Python requires password checks to always be nested — it's a syntax rule
BTo save lines of code — nested if is always shorter than two separate ifs
CSo the password is only checked when the username is correct — checking a password for an unknown user is pointless and gives away information
DBecause two separate ifs would both run, printing two error messages at once
Written part — draw the nesting structure for the ATM example (Example 2) as a set of labelled nested boxes. Label each box with: what condition it checks, what it prints if True, and what it prints if False.
Key vocabulary for this lesson
Nested conditionAn if statement written inside the body of another if statement. Each level adds 4 spaces of indentation.
Input validationChecking that user input is acceptable (correct type, non-empty, in range) before using it in calculations.
.isdigit()A string method that returns True if every character in the string is a digit (0–9). Returns False for letters, spaces, or symbols.
Defensive codingWriting code that anticipates bad input and handles it gracefully instead of crashing.
Nesting levelThe depth of an if block. Level 1 = 4 spaces. Level 2 = 8 spaces. Level 3 = 12 spaces.
Guard clauseAn early check at the top of a block that stops execution if input is invalid — keeping the main logic clean.

Teacher notes
🚨Most common error: Misaligned indentation in nested blocks — the inner else lining up with the outer if. Python will either crash with IndentationError or silently run the wrong branch. The colour-coded visualiser is the best tool for diagnosing this on the projector.
⚠️Common confusion: Students who aren't sure which else belongs to which if. Teach the rule: an else always belongs to the nearest if at the same indentation level directly above it.
⚠️.isdigit() limitation: It doesn't handle negative numbers or floats. If a student asks "what about -5?" — great question. Say: "We're handling that in Week 6 with try/except. For now, .isdigit() is our tool for whole positive numbers." Don't go down the rabbit hole today.
💡Pacing note: The refactor comparison panel is worth 5 minutes of class discussion. Students who understand when NOT to nest write far cleaner code going forward. The rule — "use and when both conditions share the same else, use nesting when they need separate responses" — should be written on the board and left there.
