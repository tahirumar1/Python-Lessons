The hotel room analogy (5 min)
Say: "Imagine a hotel. The corridor outside rooms is global — anyone in the hotel can see the fire extinguisher in the corridor. But Room 205 is local — only the person in that room can see what's on the desk inside. If someone in Room 205 leaves the hotel, their desk disappears. Live demo — code examples
Example 1 — Local variable: born and dies inside the function
main.py
def make_greeting():
    message = "Hello, World!"   # LOCAL — only exists here
    print(message)              # works fine inside

make_greeting()

# Try to access message OUTSIDE the function
print(message)    # NameError: name 'message' is not defined
Output
Hello, World!
NameError: name 'message' is not defined
Example 2 — Two functions, completely separate local scopes
main.py
def func_a():
    x = 10              # func_a's local x
    print(f"func_a: x = {x}")

def func_b():
    x = 99              # func_b's local x — DIFFERENT variable!
    print(f"func_b: x = {x}")

func_a()
func_b()
# Same name 'x' — completely independent variables
Output — two separate x variables, no conflict
func_a: x = 10
func_b: x = 99
💡This is why scope exists: Without isolation, calling func_a() would overwrite func_b()'s x. In a large program with hundreds of functions, every function could accidentally destroy other functions' variables. Scope is protection.
Example 3 — Reading a global from inside a function (works without global keyword)
main.py
app_name = "MyApp"    # GLOBAL — defined outside all functions
version  = "1.0"

def show_header():
    # Reading globals — works without 'global' keyword
    print(f"=== {app_name} v{version} ===")

def show_footer():
    print(f"Thank you for using {app_name}.")

show_header()
show_footer()
Output
=== MyApp v1.0 ===
Thank you for using MyApp.
Example 4 — Shadowing: local hides global (the silent bug)
main.py — the shadowing trap
score = 100        # GLOBAL score

def add_bonus():
    score = 150    # LOCAL score — shadows the global!
    print(f"Inside: score = {score}")

add_bonus()
print(f"Outside: score = {score}")  # still 100 — global unchanged!
Output — the global was NEVER modified
Inside: score = 150
Outside: score = 100
🚨Shadowing is a silent bug. Python doesn't warn you. The function creates a brand new local score — a completely different variable — and assigns 150 to it. The global score = 100 sits untouched. Students who expect the global to change will be confused until they understand this rule.
Example 5 — The global keyword: when you genuinely need to modify a global
main.py
score = 0

def add_points(pts):
    """Add points to the global score."""
    global score          # declares: score refers to the GLOBAL
    score += pts           # now this modifies the global

add_points(10)
add_points(25)
add_points(5)
print(f"Final score: {score}")   # 40
Output
Final score: 40
Best practice — prefer return over global
✗ Avoid — global creates hidden coupling
total = 0

def add(x):
    global total
    total += x

add(5)
add(3)
# Hard to test — depends on global state
# Call order matters everywhere in the file
Function depends on external state — hard to reuse, test, or reason about.
✓ Prefer — return makes functions self-contained
def add(current, x):
    return current + x

total = 0
total = add(total, 5)
total = add(total, 3)
# Easy to test: add(0, 5) always gives 5
# Works the same regardless of global state
Function is pure — given the same inputs, always returns the same output. Easy to test and reuse.
🎯When global IS appropriate: Program-wide constants (app name, version, config), counters or scores that genuinely need to persist across many unrelated function calls, and small scripts where the overhead of passing everything through parameters isn't justified. The key question: "Does this variable logically belong to the whole program, or to this specific function?"

Key vocabulary for this lesson
ScopeThe region of a program where a variable is accessible. Python has local and global scope (and a few others we won't cover today).
Local variableA variable created inside a function. Only visible inside that function. Destroyed when the function returns.
Global variableA variable created outside all functions. Visible everywhere in the file. Persists for the life of the program.
global keywordDeclares that an assignment inside a function refers to the global variable, not a new local one. Required only when you want to modify a global.
ShadowingWhen a local variable has the same name as a global. The local "shadows" (hides) the global inside the function. A common source of silent bugs.
Pure functionA function whose output depends only on its inputs — no side effects, no global state. Easier to test and reuse. Prefer it over functions that modify globals.

Teacher notes
🚨The shadowing bug is the hardest: Students who write score = 150 inside a function and then check print(score) outside, expecting 150, will see 100 and be completely baffled. Run Example 4 live, let them be confused, then explain. The confusion itself is the teaching moment — don't spoil it by explaining before running.
⚠️Common mistake — UnboundLocalError: When a function both reads and assigns a variable without the global keyword, Python raises UnboundLocalError — "local variable referenced before assignment". This happens with score += 10 when there's a global score but no global score declaration. Show it live so students recognise the error message.
⚠️Don't over-teach global: The goal of this lesson is for students to understand scope, recognise shadowing bugs, and use global correctly when genuinely needed. Don't spend more than 7 minutes on the global keyword — the more important message is to prefer return.
💡Scope detective timing: The 6 scenarios in the detective activity range from easy to tricky. If time is short, cover scenarios 1–4 in class and assign 5–6 as homework. Scenario 6 (UnboundLocalError) is the most valuable — don't skip it if possible.
