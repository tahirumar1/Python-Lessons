What this project brings together

This is the most complete program students have built — a real command-line application with persistent state, multiple operations, status tracking, and sorted output. It touches every major concept from all three weeks: variables, input, f-strings, conditionals, while loops, for loops, lists, list methods, slicing, and list comprehensions.

🎯Project day philosophy: Students design their own menu labels, their own task status names, their own messages. The requirements define the behaviour — not the code. Circulate and ask questions rather than giving code. Students who struggle with the done/pending status system are encountering their first real data design challenge — let them think.
Learning objectives
  • Design and build a multi-option menu-driven app from scratch using a while True loop
  • Store tasks as strings in a list and track their status in a parallel list (or as tuples)
  • Use append, remove, pop, indexing, and list comprehensions in a real program
  • Display numbered task lists using enumerate() and formatted f-strings
  • Handle invalid input gracefully — empty task names, invalid menu choices, numbers out of range
  • Present and explain their design decisions to a partner during peer review

💡Design choice to highlight: This solution uses a list of [task_name, status] pairs (a list of lists). An alternative is two parallel lists — tasks[] and statuses[]. Both are valid. Ask students which approach they used and why. There is no single right answer — this is their first real design decision.
todo_app.py — complete teacher solution
# ============================================================
#  TO-DO LIST APP  —  Week 3 Project
# ============================================================
# Data structure: list of [task_name, done_bool] pairs

tasks = []   # each item: ["task name", True/False]

def show_tasks(task_list, title="All tasks"):
    print(f"\n── {title} ({'empty' if not task_list else str(len(task_list))+' items'}) ──")
    if not task_list:
        print("  (nothing here yet)")
        return
    for i, item in enumerate(task_list, 1):
        status = "✓ done  " if item[1] else "○ pending"
        print(f"  {i}. [{status}] {item[0]}")

# ── Main loop ────────────────────────────────────────────────
while True:
    pending = [t for t in tasks if not t[1]]
    done    = [t for t in tasks if t[1]]

    print(f"\n{'='*40}")
    print(f"  TO-DO  |  {len(pending)} pending · {len(done)} done")
    print(f"{'='*40}")
    print("  1. View all tasks")
    print("  2. Add a task")
    print("  3. Mark task as done")
    print("  4. Remove a task")
    print("  5. View pending only")
    print("  6. Quit")

    choice = input("\n  Choose (1-6): ").strip()

    if choice == "1":
        show_tasks(tasks)

    elif choice == "2":
        name = input("  Task name: ").strip()
        if not name:
            print("  ✗ Task name cannot be empty.")
        else:
            tasks.append([name, False])
            print(f"  ✓ '{name}' added.")

    elif choice == "3":
        show_tasks(tasks)
        if tasks:
            raw = input("  Task number to mark done: ")
            if raw.isdigit() and 1 <= int(raw) <= len(tasks):
                idx = int(raw) - 1
                tasks[idx][1] = True
                print(f"  ✓ '{tasks[idx][0]}' marked as done.")
            else:
                print("  ✗ Invalid number.")

    elif choice == "4":
        show_tasks(tasks)
        if tasks:
            raw = input("  Task number to remove: ")
            if raw.isdigit() and 1 <= int(raw) <= len(tasks):
                removed = tasks.pop(int(raw) - 1)
                print(f"  ✓ '{removed[0]}' removed.")
            else:
                print("  ✗ Invalid number.")

    elif choice == "5":
        show_tasks(pending, "Pending tasks")

    elif choice == "6":
        print(f"\n  Goodbye! You had {len(pending)} pending tasks.")
        break

    else:
        print("  ✗ Please choose a number from 1 to 6.")

Check for understanding — self-assessment exit ticket
In the teacher solution, why are the pending and done comprehensions placed at the top of the while True loop rather than once before it?
APython requires comprehensions to be inside loops — they can't run outside
BSo they recalculate every iteration — keeping the counts accurate after every add, remove, or mark-done action
CTo save memory — comprehensions outside loops use more RAM
DIt makes no difference — the same result would happen if they were before the loop

Teacher notes
🚨Most common structural bug: Students who use a single list of task names and try to store done/pending by checking if the name has a "✓" prefix or similar. This is fragile and breaks when tasks are renamed. Catch it in the planning phase — ask "what data structure holds the status?" before students open their editor.
⚠️Off-by-one in task numbers: Students display tasks numbered 1–N (using enumerate(..., 1)) but then access them with tasks[number - 1]. Forgetting the - 1 is extremely common. Point this out when you see it: "Your menu says task 1, but Python's index for that task is 0."
⚠️Common mistake — remove without range check: Students who call tasks.pop(int(raw) - 1) without first checking that the index is within range. If the list has 2 tasks and the user enters 5, Python raises an IndexError. The safe pattern: if raw.isdigit() and 1 <= int(raw) <= len(tasks).
💡The show_tasks() function moment: After revealing the solution, point to show_tasks() and say: "This function is called 4 times in the program. Without it, those 6 lines would be copy-pasted 4 times — that's 24 lines instead of 6. Next week, you'll learn to write these." Students who copy-pasted display code in their own solution will feel this immediately.
