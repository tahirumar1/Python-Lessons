The three-zone program layout — standard for all Python files
① Helper / utility functions — at the top
Small, focused functions that each do one thing. Defined first so main() can call them. Examples: add(a,b), validate_input(raw), format_result(n). Each is independently testable.
② main() — the orchestrator — in the middle
Calls helpers in the right order. Holds the overall logic but does NOT do the work itself. Reading main() should give you the full story in 10–15 lines, even if the program has 100 lines total.
③ Entry point — ONE line at the very bottom
main() — the single line that starts everything. Only one line runs at the global level. All other code is inside function definitions.
💡Why helpers before main(): Python reads top to bottom. If main() calls format_result(), that function must already be defined above main(). Helpers first — entry point last.

Example 1 — Functions calling functions: chained temperature converter
main.py
def c_to_f(c):
    """Celsius to Fahrenheit."""
    return c * 9 / 5 + 32

def c_to_k(c):
    """Celsius to Kelvin."""
    return c + 273.15

def report(c):
    """Print full report — calls two helpers."""
    print(f"  {c}°C = {c_to_f(c):.1f}°F = {c_to_k(c):.2f} K")

def main():
    """Run the temperature converter."""
    print("Temperature conversions:")
    for t in [0, 20, 37, 100]:
        report(t)

main()

Example 3 — Complete three-zone structured program
student_report.py
# ═══ ZONE ① — Helper functions ═══════════════════════

def calculate_average(scores):
    """Return the average of a list of scores."""
    return sum(scores) / len(scores)

def get_grade(avg):
    """Return letter grade for a given average."""
    if avg >= 90:  return "A"
    if avg >= 75:  return "B"
    if avg >= 60:  return "C"
    return "F"

def print_student(name, scores):
    """Print one student's report — calls two helpers."""
    avg   = calculate_average(scores)
    grade = get_grade(avg)
    print(f"  {name:10} | avg {avg:5.1f} | grade {grade}")

# ═══ ZONE ② — main() ═════════════════════════════════

def main():
    """Run the student report system."""
    students = [
        ("Aisha",  [85, 90, 92]),
        ("Omar",   [70, 68, 74]),
        ("Zara",   [95, 98, 100]),
    ]
    print("Student Report\n" + "-" * 35)
    for name, scores in students:
        print_student(name, scores)
    print("-" * 35)

# ═══ ZONE ③ — Entry point ════════════════════════════

main()

Activity 1 — Mini calculator with 7 functions
Activity 1 · 18 minutes · Individual
Build a menu-driven calculator structured into 7 clean functions
The most structurally complete program of the course. Every logical piece lives in its own function. Reading main() alone tells the full story in ~15 lines without reading any helper.
  1. Write 4 operation functions: add(a,b), subtract(a,b), multiply(a,b), divide(a,b) — divide returns None if b is zero
  2. Write get_numbers() — asks for two floats, returns them as a tuple (a, b)
  3. Write show_menu() — prints the 5 options (4 ops + Quit)
  4. Write format_result(a, sym, b, r) — prints the formatted equation with :.4g
  5. Write main() — while True loop: show_menu → get choice → get_numbers → call operation → format_result → loop
  6. One line at the very bottom: main()

Complete starter skeleton
calculator.py — skeleton
# ═══ ZONE ① — Operation functions ═══════════════════

def add(a, b):
    """Return a + b."""
    return ???

def subtract(a, b):
    """Return a - b."""
    return ???

def multiply(a, b):
    """Return a * b."""
    return ???

def divide(a, b):
    """Return a / b, or None if b == 0."""
    if b == 0:
        print("  ✗ Cannot divide by zero.")
        return None
    return ???

# ═══ ZONE ① — Utility functions ══════════════════════

def get_numbers():
    """Ask user for two floats. Returns tuple (a, b)."""
    a = float(input("  First : "))
    b = float(input("  Second: "))
    return a, b

def show_menu():
    """Display the operation menu."""
    print("\n  1.Add  2.Subtract  3.Multiply  4.Divide  5.Quit")

def format_result(a, sym, b, r):
    """Print formatted equation."""
    print(f"  {a} {sym} {b} = {r:.4g}")

# ═══ ZONE ② — main() ═════════════════════════════════

def main():
    """Run the calculator."""
    print("=== Mini Calculator ===")
    while True:
        show_menu()
        choice = input("  Choice: ").strip()
        if choice == "5":
            print("  Goodbye!")
            break
        if choice not in ["1","2","3","4"]:
            print("  ✗ Invalid choice.")
            continue
        a, b = get_numbers()       # tuple unpacking
        if   choice=="1": format_result(a,"+",b,add(a,b))
        elif choice=="2": format_result(a,"-",b,subtract(a,b))
        elif choice=="3": format_result(a,"×",b,multiply(a,b))
        elif choice=="4":
            r = divide(a,b)
            if r is not None: format_result(a,"÷",b,r)

# ═══ ZONE ③ — Entry point ════════════════════════════

main()

Check for understanding — exit ticket
A program has 5 functions. Only main() is called at the global level — the other 4 are called from inside functions. Why is this better than putting all logic directly in main()?
AIt runs faster — Python executes small functions more efficiently than large ones
BEach function does one clear job — making code easier to read, debug, reuse, and change without breaking other parts
CIt uses less memory — each function frees its scope immediately after returning
DIt is required — Python raises an error if main() contains more than 20 lines
Written part: Look at your calculator's main() function. How many lines does it have? Could someone read those lines and understand what the calculator does — without reading the helpers? If not, what would you move into a helper?
Key vocabulary for this lesson
Function compositionUsing the return value of one function as the argument to another — chaining them together to build complex behaviour from simple parts.
Call graphA diagram showing functions as boxes and calls as arrows. Makes a program's structure visible at a glance.
DecompositionBreaking a large problem into smaller sub-problems, each solved by its own function. The core engineering skill of this week.
Single responsibilityEach function does exactly one thing. If you can't describe it in one short sentence, it probably does too much.
Tuple unpackinga, b = get_numbers() — assigning multiple return values from a function to multiple variables in one line.
Entry pointThe single main() call at the bottom of a file that starts the program. Everything flows from this one line.

Teacher notes
🚨Most common mistake — everything in main(): Students who put operation logic directly in main() rather than calling helpers. If you see main() longer than 25 lines, ask: "What does this block do? Can you give it a name? Good — make it a function."
⚠️Tuple unpacking: a, b = get_numbers() is students' first encounter with multiple return values. Show both the one-liner and the two-step version side-by-side. Neither is wrong; the unpacking form is more Pythonic.
⚠️The divide None check: Students who pass divide()'s result directly to format_result() without checking for None will crash on zero division. The skeleton shows the correct pattern — point it out before Activity 1 starts.
💡Activity 2 timing: 7 minutes for the call graph is intentionally short — a rough sketch is enough. The value is in the thinking. Fast pairs who finish early: "Which function has the most arrows? Is that a problem?"
💡Connecting to Week 3: Point out that the to-do app from Lesson 15 already had a show_tasks() helper. Students have been using this pattern — today they're learning the name and principle behind it.
